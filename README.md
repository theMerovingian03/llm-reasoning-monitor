# TRON: Transformer Reasoning Observability Network

## Overview
TRON is a system for real-time monitoring and evaluation of reasoning steps generated by large language models (LLMs). It streams model outputs, extracts structured reasoning, and evaluates each step using a secondary monitoring model to detect logical, computational, or process-level errors.

## Research Context
This project is inspired by recent research on monitoring and auditing LLM reasoning processes rather than only validating final outputs. The core idea is that intermediate reasoning steps provide richer signals for detecting errors, inconsistencies, and adversarial behaviors. By evaluating reasoning incrementally, systems can identify failures earlier and improve reliability.

## Approach
- The target LLM generates responses with reasoning enclosed in `<think>` tags.
- A streaming pipeline captures tokens in real time.
- A step parser extracts and segments reasoning into individual steps.
- A monitor model evaluates each step using a structured prompt and schema.
- The system can interrupt generation if a critical issue is detected.

## Technologies Used
- Python (asyncio)
- FastAPI (WebSocket and REST API)
- httpx (async HTTP client)
- llama-cpp-python server (local LLM inference)
- Pydantic (schema validation)
- Regular expressions (step parsing)

## Installation

1. Clone the repository:
   ```bash
   git clone <repo_url>
   cd llm-reasoning-monitor
2. Create environment and install dependencies:

    ```bash
    uv sync
3. Place GGUF model files and update paths in .env:
    ```bash
    TARGET_MODEL_PATH=...
    MONITOR_MODEL_PATH=...
    TARGET_MODEL_URL=http://127.0.0.1:8001
    MONITOR_MODEL_URL=http://127.0.0.1:8002
    MAIN_PORT=8003
4. Start the application:

    ```bash
    uv run uvicorn app.main:app --reload --port 8003
5. Connect to WebSocket endpoint:

    ```bash
    ws://127.0.0.1:8003/ws/monitor
6. Send a request:

    ```bash
    {
    "prompt": "Solve 25 × 17 step by step"
    }
The system will:
Stream tokens from the target model
Extract reasoning steps
Evaluate each step using the monitor model
Return analysis results and interrupt if necessary

## Limitations

### Hardware Constraints
- The system is designed to run locally using CPU-based inference, which leads to high latency, especially for step-level monitoring.
- Running multiple models simultaneously (target + monitor) significantly increases CPU usage and memory consumption.
- Lack of GPU acceleration limits scalability and real-time performance.
- Larger models or longer prompts can result in slow responses and potential system instability on lower-end hardware.

### Step Parsing Dependence
- Step extraction relies on heuristic parsing (sentence boundaries, punctuation, etc.), which may fail for unstructured or unconventional reasoning outputs.
- The system assumes the presence of `<think>` tags for separating reasoning from final answers. If the model does not follow this format, parsing becomes unreliable.
- Changing the target model may introduce different reasoning styles, requiring adjustments to parsing logic.
- Inconsistent formatting or missing delimiters can lead to incorrect step segmentation or missed evaluations.

### Model Reliability
- The monitor model may produce incorrect or inconsistent evaluations, especially for ambiguous or complex reasoning steps.
- Structured JSON output is not guaranteed, requiring fallback parsing mechanisms that are less reliable.
- False negatives are possible due to conservative evaluation design, while false positives may still occur in edge cases.

### Performance Overhead
- Step-level monitoring introduces multiple additional model calls, increasing latency significantly.
- Real-time interruption depends on timely evaluation, which may not be feasible with slower inference speeds.
- The system does not currently batch or optimize monitor calls, leading to inefficiencies.

### General Limitations
- The system is tightly coupled to specific model behaviors and may require reconfiguration for different LLMs.
- No persistent state or logging of reasoning traces for long-term analysis.
- Limited handling of adversarial or malformed inputs beyond basic checks.
